## Description

Define and validate the mathematical relationship between tower damage output (at each upgrade tier) and creep HP as waves progress. Right now these two curves are authored independently — this task makes the scaling explicit, documents the design targets, and wires in derived formulas so balance stays coherent as content is added. The output is a living balance spreadsheet (JSON or CSV), validated by a test suite, and referenced by TASK-11's broader playtest pass.

## Acceptance Criteria

### Scaling Model
- [ ] Define a `waveScaling` config block (in `src/data/waveDefs.ts` or a dedicated `scalingConfig.ts`) with:
  - `creepHpBase`: base HP for a wave-1 standard creep
  - `creepHpGrowthPerWave`: multiplicative factor per wave (e.g. 1.12 = +12%/wave)
  - `creepSpeedGrowthPerWave`: additive or multiplicative speed increase per wave
  - `bossHpMultiplier`: multiplier applied on top of wave HP for boss creeps (see TASK-013)
- [ ] Document the "time-to-kill" (TTK) target for each wave: an un-upgraded tower of each type SHOULD kill a single standard creep of that wave in N shots/seconds (define N per tower type)
- [ ] Verify that a **fully-upgraded single-path tower** can kill that wave's standard creep in ≤ 50% of the TTK of an un-upgraded tower at the same wave
- [ ] Verify that a level-1 (un-upgraded) tower placed on wave 1 is NOT still one-shotting wave-10 creeps (tower output must not dramatically outpace creep HP growth at any wave)

### Tower Damage Tiers
- [ ] Each tower type defines a `baseDamage` in its `TowerDef`
- [ ] Each upgrade tier defines a `damageMultiplier` or `damageAddend` in its `UpgradeDef`
- [ ] A utility function `towerEffectiveDPS(towerDef, upgradeState, wave): number` computes combined DPS for a given state (used in tests and debug overlay)
- [ ] A utility function `creepEffectiveHP(wave, creepType): number` computes current HP for any wave (used in tests and balance spreadsheet export)

### Balance Targets (to be tuned, not hard-coded)
- [ ] Wave 1: a single un-upgraded Cannon tower SHOULD kill a standard creep before it reaches the midpoint of the path
- [ ] Wave 10: a player with 3–4 mixed un-upgraded towers SHOULD survive (some creep escapes expected but not a wipe)
- [ ] Wave 15: survival requires at least Tier-2 upgrades on primary towers
- [ ] Wave 20: survival requires a mix of Tier-3+ upgrades AND at least one complementary roguelike offer active
- [ ] Boss waves (5, 10, 15, 20): the boss SHOULD survive at least 50% of the path against a player who spent zero gold on upgrades (bosses must not be trivialized by raw DPS alone)

### Validation
- [ ] Automated test: run `towerEffectiveDPS` × path traversal time vs `creepEffectiveHP` for waves 1, 5, 10, 15, 20 — assert results land in the defined target bands
- [ ] Export a human-readable balance table: rows = wave, columns = tower type × upgrade tier, values = kill-time in seconds; output as `src/data/balance-table.json` (generated, not hand-authored)
- [ ] Balance table is regenerated by running `npm run balance` (add script to package.json)

### Debug Tooling
- [ ] Add an in-game debug overlay (toggled by pressing `B` in development builds) showing live DPS for the selected tower and live HP of the nearest creep
- [ ] Debug overlay is stripped from production builds (guarded by `import.meta.env.DEV`)

## Notes

This task deliberately separates the scaling math from the content tuning in TASK-11. TASK-014 establishes the model and tests; TASK-11's playtest sessions tune the coefficients within that model.

The boss HP formula (`creepEffectiveHP × bossHpMultiplier`) should be cross-checked against TASK-013's boss archetype definitions to ensure bosses don't become unhittable at high waves.
